# 实时检测性能分析报告

## 问题描述

用户反馈："实时捕获屏幕已经启用了，但是页面太卡了，说明我们推理太慢了，我们需要对画面进行压缩到640*640"

从日志中可以看到：
- 屏幕分辨率：2560x1600 (约 410 万像素)
- 检测帧率：30 FPS
- 使用的模型：yolo11n.onnx

## 当前架构分析

### 数据流

```
屏幕截图 (2560x1600)
    ↓ xcap::Monitor::capture_image()
原始 RGBA 图像 (~15.7 MB/帧)
    ↓ resize_exact(640, 640)
推理图像 (640x640, ~1.2 MB)
    ↓ YOLO 推理
检测结果
    ↓ draw_boxes() + encode_image_fast()
编码图像 (~50-100 KB)
    ↓ Base64 编码 (~65-130 KB)
网络传输
    ↓ WebSocket/HTTP
前端渲染
```

### 性能瓶颈分析

#### 1. **Base64 编码 - 主要瓶颈** ⚠️
```rust
// 当前代码
Ok(BASE64.encode(buffer.into_inner()))
```

**问题**：
- 即使图像已经压缩到 640x640，Base64 编码仍会增加 **33%** 的数据大小
- 1MB 的 JPEG 图像 → 1.33MB 的 Base64 字符串
- 30 FPS × 1.33MB = **40 MB/s** 的数据传输

**解决方案**：
- 使用二进制传输（WebSocket）
- 或者使用更激进的 JPEG 压缩（质量 40-50）

#### 2. **JPEG 编码质量过高** ⚠️
```rust
// 当前代码
let quality = if rgb.width() <= 640 { 70 } else { 60 };
```

**问题**：
- 70% 质量对于实时预览来说太高
- 高质量 JPEG 编码速度慢 3-5 倍

**建议**：
- 质量降到 40-50
- 可以根据网络状况动态调整

#### 3. **帧率太高** ⚠️
```rust
// 当前代码
let frame_duration = Duration::from_secs_f64(1.0 / fps_limit as f64);
// fps_limit = 30
```

**问题**：
- 30 FPS 对于实时检测来说太高
- 人类眼睛只能感知约 24 FPS 的流畅度
- 移动物体检测不需要高帧率

**建议**：
- 降低到 10-15 FPS
- 或者添加智能帧率控制（动态调整）

#### 4. **帧积压问题** 📊
**问题**：
- 如果某帧处理时间 > 33ms，后续帧会积压
- 积压越来越多，导致越来越卡

**建议**：
- 添加帧跳过机制
- 如果当前帧处理超时，直接跳过

#### 5. **前端渲染性能** 📊
**问题**：
- 每秒 30 次 `src = "data:image/jpeg;base64,..."` 更新
- 频繁的 DOM 操作导致 reflow/repaint
- 图像解码也需要时间

**建议**：
- 使用 Canvas 渲染
- 添加渲染节流

## 优化方案

### 方案 1：快速优化（立即见效）

#### 1.1 降低帧率和质量
编辑 `src-tauri/src/modules/yolo/services/desktop_capture.rs`：

```rust
// 第 476 行附近 - 修改帧率
let frame_duration = Duration::from_secs_f64(1.0 / 15.0); // 从 30 降到 15 FPS

// 第 364 行附近 - 降低 JPEG 质量
let quality = 40; // 从 70/60 降到 40
```

#### 1.2 添加帧跳过机制
在循环开始处（第 490 行附近）：

```rust
loop {
    // 性能优化：帧跳过
    // 如果上一帧还没处理完，跳过当前帧
    static mut LAST_FRAME_TIME: std::time::Instant = std::time::Instant::now();
    unsafe {
        let elapsed = LAST_FRAME_TIME.elapsed();
        if elapsed < frame_duration {
            // 还没到时间，跳过
            continue;
        }
        LAST_FRAME_TIME = std::time::Instant::now();
    }
    
    // 原有的捕获逻辑...
}
```

### 方案 2：激进优化（需要前端配合）

#### 2.1 使用二进制 WebSocket 传输
修改 Rust 端：
```rust
// 使用 bytes crate 进行二进制传输
use bytes::{Bytes, BytesMut};

// 编码为 JPEG 后直接发送 bytes，而不是 Base64
let jpeg_bytes = buffer.into_inner();
app.emit("desktop-capture-frame-binary", jpeg_bytes);
```

修改前端：
```typescript
// 使用 ArrayBuffer 接收
socket.on('desktop-capture-frame-binary', (data: ArrayBuffer) => {
    const blob = new Blob([data], { type: 'image/jpeg' });
    const url = URL.createObjectURL(blob);
    img.src = url;
    URL.revokeObjectURL(url);
});
```

#### 2.2 前端 Canvas 渲染
```typescript
const canvas = document.getElementById('canvas') as HTMLCanvasElement;
const ctx = canvas.getContext('2d');

// 渲染优化
let lastRenderTime = 0;
const targetFPS = 15;

function renderFrame(base64Data: string) {
    const now = performance.now();
    if (now - lastRenderTime < 1000 / targetFPS) {
        return; // 跳帧
    }
    
    const img = new Image();
    img.onload = () => {
        ctx.drawImage(img, 0, 0);
        lastRenderTime = now;
    };
    img.src = 'data:image/jpeg;base64,' + base64Data;
}
```

### 方案 3：智能优化（高级）

#### 3.1 动态帧率控制
```rust
// 根据实际处理时间动态调整帧率
let mut current_fps_limit = fps_limit as f32;
let mut avg_process_time = 0.0f32;

loop {
    let frame_start = Instant::now();
    
    // 处理帧...
    
    let elapsed = frame_start.elapsed().as_secs_f32();
    avg_process_time = avg_process_time * 0.9 + elapsed * 0.1;
    
    // 动态调整：如果处理时间 > 帧间隔，降低帧率
    if avg_process_time > 1.0 / current_fps_limit {
        current_fps_limit = (current_fps_limit * 0.9).max(5.0);
    } else if avg_process_time < 1.0 / current_fps_limit * 0.7 {
        current_fps_limit = (current_fps_limit * 1.1).min(fps_limit as f32);
    }
    
    thread::sleep(Duration::from_secs_f32(1.0 / current_fps_limit - elapsed));
}
```

#### 3.2 自适应质量控制
```rust
let mut current_quality = 70u8;
let mut frame_size_avg = 0u32;

loop {
    // 处理...
    
    let frame_size = encoded.len();
    frame_size_avg = frame_size_avg * 0.9 + frame_size as u32 * 0.1;
    
    // 动态调整质量以维持目标带宽
    let target_frame_size = 50_000u32; // 50KB 目标
    if frame_size_avg > target_frame_size {
        current_quality = (current_quality - 5).max(30);
    } else if frame_size_avg < target_frame_size * 0.8 {
        current_quality = (current_quality + 5).min(80);
    }
}
```

## 推荐的优化顺序

1. **立即优化**（修改 2 行代码）：
   - 降低帧率到 15 FPS
   - 降低 JPEG 质量到 40

2. **快速优化**（添加帧跳过）：
   - 防止帧积压
   - 改善卡顿感

3. **中期优化**（修改前端）：
   - 使用 Canvas 渲染
   - 添加渲染节流

4. **长期优化**（架构调整）：
   - WebSocket 二进制传输
   - 动态帧率/质量控制

## 结论

当前系统卡顿的主要原因是：
1. **帧率过高**（30 FPS）
2. **图像质量过高**（70%）
3. **Base64 编码开销**
4. **前端渲染频率过高**

通过降低帧率和质量，可以立即获得 **3-5 倍** 的性能提升。
如果需要进一步优化，建议使用 WebSocket 二进制传输。
